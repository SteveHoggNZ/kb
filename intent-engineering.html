Intent Engineering
Workshop Guide

Stop Crafting Prompts.
Start Engineering Intent.

Prompt engineering asks "what words make the AI do the right thing?" Intent engineering asks "how do I make the right thing the only possible outcome?"

The Idea

What is Intent Engineering?

Intent engineering is the practice of communicating what you want to achieve — the goal, constraints, success criteria, and context — rather than obsessing over the exact phrasing of a prompt. It shifts the focus from words to outcomes.

As AI models have become more capable, the bottleneck is no longer "did I use the magic words?" It's "do I actually know what I want, and have I structured things so the AI can't get it wrong?"

🎯

Goals, Not Scripts

Define what success looks like — measurable criteria the AI can work toward — rather than scripting every step.

🧱

Structure, Not Hope

Encode your intent into structural mechanisms (tests, types, linters) that make wrong behaviour impossible, not just discouraged.

📦

Context, Not Repetition

Give the AI the right information at the right time. Compress context into dense handoff artefacts rather than dumping everything.

Core Insight

Physics Thinking

The central idea of intent engineering is distinguishing between laws and physics. Laws are instructions — they can be broken. Physics are structural constraints — they can't.

Law (Prompt Instruction) Physics (Structural Enforcement)
"Don't break existing behaviour" Playwright E2E tests that fail if behaviour changes
"Follow the existing coding style" Linter configured to enforce style rules
"Make sure the cart logic is correct" Unit tests that assert cart calculations
"Don't introduce type errors" Compiler that refuses to build

The rule: If a human has to manually check something, the harness is incomplete. Encode every compliance check as an automated constraint.

From Prompts to Intent

Prompt engineering is a subset of intent engineering. Here's what changes when you think bigger.

Comparison

What's Actually Different?

Prompt engineering treats the prompt as the product. Intent engineering treats the entire system — prompts, structure, feedback loops, constraints — as the product.

Prompt Engineering

  • Focus on exact wording and formatting
  • Tricks, templates, and magic phrases
  • One prompt → one response
  • Hope the AI follows instructions
  • Retry with different words when it fails
  • All context in a single prompt

Intent Engineering

  • Focus on goals, constraints, and success criteria
  • Structural enforcement through tooling
  • Multi-phase workflows with handoff artefacts
  • Make wrong behaviour structurally impossible
  • Feedback loops that teach the AI what failed
  • Information diets — right context at right time
Layers

Intent Operates at Multiple Levels

A single prompt is just one layer. Intent engineering stacks mechanisms so each layer catches what the one above misses.

Vision

North-star goal and measurable success criteria. What does "done" look like?

Strategy

The reasoning behind your approach. Why this methodology? What trade-offs are you making?

Constraints

Explicit boundaries — what the AI must not do. Negative instructions that define the safe operating space.

Execution

Structured agent prompts with output shape, required sections, and detail level defined.

Feedback

Harnesses (compiler, tests, linters) that enforce correctness and teach the AI what went wrong.

Key Insight

Prompts Are Suggestions. Harnesses Are Guarantees.

A prompt says "please don't break the cart." A test suite says "the cart works or you don't proceed." Both communicate intent, but only one enforces it.

The best intent engineering systems use prompts for guidance and harnesses for enforcement. The prompt sets direction; the harness sets boundaries.

Think of it this way: Prompt engineering is writing a good brief. Intent engineering is writing a good brief, hiring a QA team, setting up CI/CD, and defining acceptance criteria — all encoded into the system the AI operates within.

The RPI Methodology

Research → Plan → Implement. Three phases, each in a fresh context, connected by dense handoff artefacts.

Why Phases?

Decompose → Route → Recompose

AI agents have limited context windows. If you ask one to research, plan, and implement all in the same session, the early context gets pushed out as implementation details pile up. RPI solves this by splitting the work into phases, each operating in a fresh context with only the artefacts it needs.

🔍

Phase 1: Research

The agent explores the codebase freely. No implementation, no planning — pure discovery. Produces a dense research.md with file paths, line numbers, and observations.

📐

Phase 2: Plan

A fresh agent reads only research.md. It produces a step-by-step implementation plan with interface definitions, risk areas, and test strategies.

⚡

Phase 3: Implement

A fresh agent reads only plan.md. Executes each step, running the full test suite after every change. Doesn't proceed until green.

Information diets: Each phase gets only what it needs. The Plan agent never sees raw code — only the compressed research. The Implement agent never sees the research — only the ordered plan. This prevents context waste and forces density.

The Artefacts

Handoff Documents Are the Connective Tissue

Each phase produces a handoff artefact that carries intent forward. These aren't casual notes — they're dense, structured documents that compress hours of exploration into something a fresh agent can act on immediately.

research.md structure 
## 1. Entry Point (UI)
   File paths, button click handlers, form submissions

## 2. Controller Layer
   Route → controller → method, with line numbers

## 3. Service Layer
   Business logic, dependencies, cross-cutting concerns

## 4. Data Access Layer
   Repository patterns, EF queries, stored procedures

## 5. Observations
   Refactoring opportunities, code smells, coupling points
plan.md structure 
## Implementation Steps
   Ordered, atomic steps — each one independently testable

## Interface Definitions
   New interfaces and their method signatures

## DI Registration
   Exactly what to register, in which order

## Risk Register
   Risk → Severity → Which harness catches it → Specific test
Three Principles

What Holds It Together

1

Decompose-Route-Recompose

Break complex work into phases, route each to a fresh context with exactly the artefacts it needs, then recompose the results. This works with context window limitations instead of fighting them.

2

Physics Thinking

Don't tell the agent what to do — make wrong things impossible. A failing test is worth a thousand prompt instructions. Structural constraints are your most reliable form of intent communication.

3

Harnesses Matter More Than Models

The test suite, the compiler, the linter — these are your real safety net. A mediocre model with a great harness outperforms a brilliant model with no guardrails. Invest in the harness first.

Intent Engineering Patterns

Concrete, reusable techniques for encoding intent into your AI workflows.

Pattern 1

Intent Scaffolding

Don't just tell the agent what to do. Define the shape of the output: required sections, detail level, format, and output location. The agent fills in the scaffold rather than guessing what you want.

Vague

  • "Research how the shopping cart works in this codebase"

Scaffolded

  • "Trace the add-to-cart flow from UI to database"
  • "For each layer, identify files, methods, and line numbers"
  • "Produce research.md with sections: Entry Point, Controller, Service, Data Access, Observations"
  • "Save to docs/adc/extract-cart-service/research.md"

The scaffolded version communicates the same intent, but the agent knows exactly what shape the answer should take, what detail level is expected, and where to put the result.

Pattern 2

Constraint Systems

Explicit negative instructions that define the safe operating space. Constraints are often more useful than positive instructions because they prevent the most common failure modes.

Example constraints 
## Constraints
- Do NOT modify the database schema
- Do NOT change the plugin architecture
- Do NOT proceed to the next step until all tests pass
- Focus only on the shopping cart steel thread

Each constraint prevents a specific, known failure mode.
They're action-negative: defining what NOT to do.
Pattern 3

Risk Registers as Intent

Predict what could go wrong and map each risk to the specific harness that catches it. This transforms "be careful" into "here's exactly which test will fail if you deviate."

Risk Severity What Catches It
Circular DI dependency Critical App fails to start → all Playwright tests fail
Missing DI registration Critical InvalidOperationException → any E2E test
Cart total miscalculation High Unit test asserting cart math
Broken checkout button High Playwright test clicking "Add to cart"

The risk register is a form of intent: it tells the implementing agent where the danger zones are and gives it confidence that mistakes will be caught.

Pattern 4

Feedback Loops as Self-Correction

Instead of hoping the agent gets it right the first time, build mandatory checkpoints into the workflow. The error message itself becomes a remediation instruction.

Feedback loop 
After EACH step:
  1. dotnet format --verify-no-changes
  2. dotnet test
  3. npx playwright test

If a test fails:
  → Read the full error output
  → Diagnose the root cause
  → Fix the issue
  → Re-run until green
  → Only then proceed to the next step

Why this works: Test failures are precise, specific feedback. "Expected 42.99, got 0" is infinitely more useful than "please make sure the cart logic is correct." The harness teaches the agent what's wrong in machine-readable terms.

Pattern 5

Information Diets

Give each agent phase only the information it needs — nothing more. This prevents context window waste and forces the output of each phase to be dense and self-contained.

🔍

Research Agent Sees

Full codebase access. The goal is to explore freely and compress findings into research.md.

📐

Plan Agent Sees

Only research.md. No raw code. Forces it to plan from the compressed summary, not get lost in details.

⚡

Implement Agent Sees

Only plan.md + codebase access. Follows the plan step-by-step. Doesn't re-research, doesn't re-plan.

Intent Engineering In Practice

A workshop applying these patterns to refactor a real-world .NET e-commerce codebase with AI agents.

The Challenge

Refactoring nopCommerce with AI

nopCommerce is a large, real-world .NET e-commerce platform. The workshop demonstrates that AI coding agents can meaningfully refactor legacy code using a disciplined methodology — and that the methodology matters more than the model.

The "steel thread" is the shopping cart flow: from clicking "Add to cart" in the UI, through controllers and services, down to the database and back. One narrow, vertical slice through the entire architecture.

The Exercises

Five Steps, Building on Each Other

0. Get Orientated

Understand the codebase, the steel thread, and the key files. No AI agents yet — just context-setting for the humans.

1. Build the Harness

Write Playwright E2E tests and unit tests before any refactoring begins. The harness defines "correct" structurally. This is the single most important step.

2. Research the Steel Thread

Agent explores the add-to-cart flow end-to-end. Produces a dense research.md with every file, method, and connection point documented.

3. Plan the Refactoring

Fresh agent reads only research.md. Produces an ordered implementation plan with interface definitions, DI registration, and a risk register.

4. Implement with Feedback Loops

Fresh agent follows the plan step-by-step, running the full test suite after each change. The harness catches mistakes; the agent learns from the errors.

The Wow Moments

When Physics Thinking Pays Off

Three moments in the workshop where you see intent engineering working in real time:

🔴

The Compiler Catches a Type Error

The agent introduces a subtle type mismatch. The C# compiler refuses to build. The agent reads the error, fixes the type, and moves on. No human intervention. Physics thinking.

🧪

A Unit Test Catches a Logic Regression

The agent refactors the cart service and accidentally changes calculation order. The unit test fails with "Expected 42.99, got 38.50." The agent diagnoses and fixes. Harnesses over models.

🎭

Playwright Catches a UI Regression

The extracted service works, but a missing DI registration means the page throws a 500 error. Playwright catches it by trying to click "Add to cart." End-to-end verification. All three principles working together.

Success Criteria

How We Know It Worked

The vision document defines seven measurable success criteria. Intent engineering means defining these up front, not after the fact:

✓ Research before code — agent explored before touching anything
✓ Plan before implementation — ordered steps with risk analysis
✓ Harnesses before refactoring — tests existed before changes
✓ Feedback loops — test suite ran after every change
✓ Dense handoffs — research.md and plan.md carried intent forward
✓ Physics enforcement — compiler + tests caught real errors
✓ No manual checking — if the harness is green, the refactoring is correct

The takeaway: The model doesn't matter as much as the methodology. A well-structured intent engineering system — with phases, harnesses, and feedback loops — produces reliable results regardless of which AI model you use.
